01 / STLC

Software Testing Life Cycle

The STLC is a systematic process that verifies whether software quality meets expectations — a fundamental part of the SDLC.

01
Requirement Analysis
Objective: Understand what needs to be tested
  • Review Software Requirements Document (SRD)
  • Interview stakeholders to gather information
  • Identify ambiguities and inconsistencies
  • Identify potential risks impacting testing
Requirements Traceability Matrix (RTM)
02
Test Planning
Objective: Create overall test strategy and plan
  • Identify testing objectives and scope
  • Develop test strategy and methods
  • Identify environment and resources needed
  • Estimate time and cost
  • Assign roles and responsibilities
Test Plan Document
03
Test Case Development
Objective: Design detailed test cases
  • Write clear, concise test cases
  • Create test data and scenarios
  • Identify expected results for each case
  • Review and validate test cases
  • Update RTM
Test Cases, Scripts, Test Data
04
Test Environment Setup
Objective: Define hardware, software, and network conditions
  • Install and configure required software and tools
  • Set up servers, browsers, OS, and devices
  • Prepare access credentials and permissions
  • Validate environment before execution
Test Environment Ready
05
Test Execution
Objective: Execute prepared test cases
  • Run manual or automated test cases
  • Log defects with severity and priority
  • Perform defect retesting after fixes
  • Perform regression testing if required
  • Collect and analyze test results
Execution Reports, Defect Reports
06
Test Closure
Objective: Complete and document all testing activities
  • Prepare Test Summary Report
  • Ensure all defects are tracked and closed
  • Clean up test environment
  • Archive cases, data, and reports
  • Conduct retrospective for lessons learned
Test Closure Report, Lessons Learned

02 / Manual Testing

Manual Testing

Manual testing involves a person actively using software to find bugs and issues without automation tools — following predefined test cases.

Black Box
Tests functionality without knowledge of internal code structure.
Functional Testing
Verifies features work according to specified requirements and checks if the system does what it's supposed to do.
Non-Functional Testing
Focuses on performance, usability, and quality aspects — not directly related to specific functions.
Regression Testing
Done after code changes or bug fixes to verify new code doesn't negatively impact existing features.
White Box
Tests the internal structure of the system — designs and coding.
Path Testing
Checks all possible execution paths and ensures each function behaves as expected.
Security Testing
Finds security issues in code using static analysis and secure development best practices.
Loop Testing
Checks that loops operate correctly and efficiently, preventing infinite loops or logic flaws.
Data Flow Testing
Tracks flow of variables through the program — declaration, initialization, and usage.
Gray Box
A combination of both white-box and black-box testing approaches.
Partial Knowledge
Internal structure is partially known to the tester — provides a balanced view.
Dual Focus
Tests both internal structure and functionality simultaneously for comprehensive coverage.
10-Step Process
STEP 01
Understand Project Requirements
Study functional and non-functional requirements, review user stories, and gather stakeholder input.
STEP 02
Create Test Plan
Define scope (what to test, what to exclude), set objectives, list resources, and specify environments.
STEP 03
Design Test Cases
  • Test ID, Description, Preconditions
  • Step-by-step execution instructions
  • Expected Result, Postconditions
STEP 04
Set Up Test Environment
Install applications, configure settings, ensure access to test data, and verify hardware/software requirements.
STEP 05
Execute Test Cases
Follow test steps, record actual results, mark pass/fail status, and note observations.
STEP 06
Log and Report Defects
  • Defect ID, Summary, Steps to reproduce
  • Actual vs. expected results
  • Severity (critical / major / minor), screenshots, logs
STEP 07
Track and Verify Defects
Monitor defect status, retest fixes, and update status (closed or reopened).
STEP 08
Conduct Regression Testing
Ensure defect fixes or changes haven't broken existing functionality.
STEP 09
Prepare Test Closure Reports
Summarize testing activities, test results (passed/failed), defects found/resolved, and lessons learned.
STEP 10
Provide Feedback and Recommendations
Analyze software quality, suggest process improvements, and recommend future testing strategies.
Advantages & Disadvantages

✓ Advantages

  • Fast and accurate visual feedback — detects dynamically changing GUI bugs
  • Less expensive — no high-level skills or specific tools required
  • No coding required — easy to learn for new testers
  • Efficient for unplanned changes — adapts easily
  • Human intuition — testers can use judgment and creativity
  • Early testing — can start before full development

✗ Disadvantages

  • Less reliable — doesn't cover all testing aspects
  • Cannot be reused — separate cases needed for each software
  • Large human resources required
  • Needs experience — testers must know the application well
  • Time-consuming — large projects take significant time
03 / Automation Testing

Automation Testing

Automation testing uses special software tools to execute tests, writing scripts to test software automatically — reducing (not eliminating) manual effort.

Automation Types

Type Description
Unit Testing Tests the smallest piece of code logically isolated in a system
Integration Testing Tests individual components combined as a group
Smoke Testing Preliminary check to determine if software build is stable
Performance Testing Determines system performance under a particular load
Regression Testing Confirms software still works after changes or fixes
Security Testing Uncovers risks and vulnerabilities in the system
Acceptance Testing Determines end users' approval of the software
API Testing Validates API functionality, security, and reliability
UI Testing Ensures all screen elements function as desired
Automation Frameworks
FRAMEWORK 01
Linear Framework
Most basic form — record and playback. Testers create and execute scripts for each test case. Suitable for small teams with limited automation experience.
FRAMEWORK 02
Modular-Based Framework
Organizes test cases into small independent modules, each with different scenarios. All modules handled by a master script — requires pre-planning.
FRAMEWORK 03
Library Architecture
An expansion of the modular framework — tasks grouped into functions by common objective, stored in libraries for quick access and greater reusability.
FRAMEWORK 04
Data-Driven Framework
Test data and scripts separated — allows testing with multiple data sets without changing scripts. Increases test coverage efficiently.
FRAMEWORK 05
Keyword-Driven Framework
Uses keywords to describe actions. Non-technical team members can write tests using action keywords without coding knowledge.
FRAMEWORK 06
Hybrid Framework
Combines multiple frameworks to leverage benefits of each. Most commonly used in real-world projects for maximum flexibility.
Advantages & Disadvantages

✓ Advantages

  • Simplifies test case execution — can run unattended
  • Improves reliability — equal focus on all testing areas
  • Increases test coverage — more cases can be executed
  • Minimizes human interaction — no human error from neglect
  • Saves time and money in the long run
  • Earlier defect detection — documents defects for easier fixing

✗ Disadvantages

  • High initial cost — requires time and money investment upfront
  • 100% automation not possible — some tests require human intervention
  • Can't test user-friendliness, graphics, or sound
  • Programming knowledge required to write test scripts
  • False positives and negatives — tests may incorrectly pass or fail
04 / Comparison

Manual vs Automation

Choosing the right approach depends on project context. Here's a comprehensive comparison to guide your decision.

Parameter Manual Testing Automation Testing
Test Execution Executed manually by QA testers Executed automatically using tools and scripts
Efficiency Less efficient and time-consuming More efficient and less time-consuming
Test Coverage Difficult to ensure 100% coverage More test coverage can be ensured
Programming Knowledge Not required — only product knowledge needed Required to write test scripts
Reliability Less reliable due to human error More reliable — performed by tools/scripts
Investment Heavy investment in human resources Investment in tools rather than human resources
Regression Testing May not catch all regression bugs Effective for catching bugs due to code changes
Cost Less expensive initially Higher initial investment
Reusability Cannot be reused easily Can be reused for multiple test cycles
Human Resources Requires numerous human resources Requires fewer human resources
When to Use Each

Use Manual Testing

  • Exploratory testing
  • Usability testing
  • Ad-hoc testing
  • New feature testing (one-time)
  • UI/UX testing
  • Small projects with limited budget
  • Testing requiring human judgment

Use Automation Testing

  • Regression testing
  • Load/Performance testing
  • Repetitive test cases
  • Data-driven testing
  • Large projects with frequent releases
  • Smoke testing
  • Cross-browser/platform testing
  • Agile/DevOps environments

GUI-based automation using screenshots — automate anything visible on screen.
Appium
Open-source tool for mobile app testing on iOS, Android, and Windows platforms.
JMeter
Open-source load testing for functional behavior and performance measurement.
06 / Best Practices

Best Practices

Proven practices that separate professional testing from ad-hoc efforts.

Automation Best Practices

  • Plan self-contained, clearly defined test cases
  • Plan execution order — one test creates state for the next
  • Use tools with automatic scheduling capabilities
  • Set up alarms for test failures to decide continue/abort
  • Continuously reassess test plans as application evolves

Manual Best Practices

  • Understand requirements thoroughly before starting
  • Create comprehensive test cases — positive and negative
  • Maintain and update test documentation throughout
  • Perform regression testing after every change
  • Communicate effectively with regular team updates

Conclusion

The Balanced Approach

Effective software testing requires combining both manual and automation methodologies. The STLC provides a structured framework, while knowing when to use each approach optimizes resources and improves software quality.

  • Follow STLC phases systematically
  • Use manual testing for exploratory and usability scenarios
  • Implement automation for regression and repetitive tests
  • Select tools based on project requirements and budget
  • Continuously improve through retrospectives
  • Maintain comprehensive documentation throughout
  • Communicate effectively with all stakeholders
