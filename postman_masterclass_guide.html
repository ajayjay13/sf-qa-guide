01
What is Postman?
Understanding the tool and its ecosystem

Postman is a tool that allows you to easily work with APIs by building and sending HTTP requests to servers. It started as a Chrome extension but evolved into a full-featured standalone platform for API development, testing, and collaboration.

🚀
API Testing
Send HTTP requests, inspect responses, and validate API behavior manually or automatically.
⚡
Automation
Write JavaScript tests that run after each response. Automate entire test suites with the Collection Runner.
🤝
Collaboration
Share collections, fork workspaces, and create pull requests — like GitHub for API collections.
🔧
CI/CD Integration
Run collections in pipelines using Newman CLI or the Postman CLI with GitHub Actions, Jenkins, and more.
Deprecated: Chrome Extension Do NOT use the Google Chrome extension for Postman — it is no longer updated and lacks modern features. Use the standalone app or the web version at go.postman.co.

What Postman is NOT

  • ❌ Not a UI testing tool — can't click buttons or fill out forms
  • ❌ Not a performance/load testing tool
  • ⚠️ Not designed for security testing (though possible)
  • ✅ Perfect for functional API testing and automation

Your First Request
Building and sending HTTP requests

Work in Postman is organized in Workspaces. Each workspace can contain collections of requests.

1
Create a Workspace
Go to Workspaces → Create Workspace. Choose Personal for solo work or Team for collaboration.
2
Open a New Request Tab
Click the + button to open a new request. Select the HTTP method and enter the URL.
3
Send the Request
Click Send. View the response body, headers, and status code in the response panel below.
4
Check the Status Code
A 200 OK or any 2XX code means success. Read the API documentation to know what to expect.
04
HTTP Protocol
How requests and responses work

All APIs use HTTPS — Hypertext Transfer Protocol Secure. HTTPS ensures the connection is encrypted via SSL/TLS. From a testing perspective, HTTP and HTTPS behave identically.

📤 HTTP Request
  • URL The endpoint address
  • Method GET, POST, PUT, PATCH, DELETE
  • Headers Metadata like Authorization
  • Body Data payload (for POST, PATCH)
📥 HTTP Response
  • Status 200, 201, 400, 404, 500...
  • Headers Content-Type, Cache-Control...
  • Body JSON, XML, HTML, plain text
05
Collections & Variables
Organizing requests and managing dynamic data

A Collection is a group of saved API requests. Typically you'll have one collection per API. Collections can be shared, forked, versioned, and run as a complete test suite.

Using Variables

Variables let you avoid repeating the same values and make changes centrally. Reference them with double curly braces: {{variableName}}

URL with Variables 
// Base URL stored as collection variable: baseUrl = https://simple-books-api.click
{{baseUrl}}/books              → https://simple-books-api.click/books
{{baseUrl}}/books?type=fiction   → filtered request
{{baseUrl}}/books/:bookId        → path variable

Initial Value vs Current Value

📤 Initial Value
Shared when you export or publish the collection. Never put secrets here — it's visible to anyone with access to the collection.
🔒 Current Value
Private to you. This is the value Postman actually uses when running requests. Safe for tokens and sensitive data.

Query Parameters

Query parameters start after ? in the URL and are formatted as key=value pairs joined with &.

Query Parameters 
// Single parameter
{{baseUrl}}/books?type=fiction

// Multiple parameters (joined with &)
{{baseUrl}}/books?type=non-fiction&limit=5

// Path variable (bookId is dynamic)
{{baseUrl}}/books/:bookId   → e.g. /books/3
06
HTTP Methods
GET, POST, PATCH, DELETE and when to use them
Method Purpose Has Body? Typical Response
GET Retrieve data No 200 OK
POST Create new resource Yes (JSON) 201 Created
PATCH Partial update to existing resource Yes (JSON) 204 No Content
DELETE Delete a resource No (empty) 204 No Content
JSON Body Format For POST and PATCH requests, set Body → Raw → JSON. Use double quotes for strings, no quotes for numbers and booleans, and commas between key-value pairs.
JSON Request Bodies 
// POST /orders — Create a new order
{
  "bookId": 1,
  "customerName": "John Doe"
}

// PATCH /orders/:orderId — Update customer name
{
  "customerName": "Jane Doe"
}

// POST /api-clients — Register for a token
{
  "clientName": "MyClient",
  "clientEmail": "me@example.com"
}
07
Authentication
API tokens, Bearer auth, and managing credentials

Some API endpoints are public (no auth needed). Protected endpoints require an access token — a temporary password generated by the API after registration.

1
Register an API Client
Send POST /api-clients with your clientName and clientEmail. The response contains your access token.
2
Store the Token Securely
Save the token in the Current Value of a collection variable (e.g., accessToken). Never put it in Initial Value.
3
Add Authorization Header
In the Authorization tab, choose Bearer Token and enter {{accessToken}}. Or add manually: Authorization: Bearer {{accessToken}}
Authorization Header 
// Manual header format
Authorization: Bearer {{accessToken}}

// Status 409 = already registered, use a different email
// Status 401 = missing or invalid token
// Token is valid for 7 days

HTTP Status Codes
Understanding what the server is telling you
200
OK
Request succeeded. Data returned.
201
Created
Resource created successfully.
204
No Content
Success, but no body (e.g., DELETE, PATCH).
400
Bad Request
Invalid input from the client side.
401
Unauthorized
Missing or invalid authentication token.
404
Not Found
Resource doesn't exist or was deleted.
409
Conflict
Resource already exists (e.g., registered email).
429
Too Many Requests
Rate limit exceeded. Slow down.
500
Server Error
Something went wrong on the server.

Writing API Tests
Automating response validation with JavaScript

Tests in Postman are written in JavaScript and run after the response arrives. They live in the Tests tab of any request. Postman uses the Chai.js assertion library.

Golden Rule: Make Tests Fail First Before trusting a test, verify it fails when the assertion is wrong. Change the expected value to something incorrect, run it, confirm it fails, then fix it. This ensures the test actually works.
JavaScript · Tests Tab 
// Test 1: Check status code is 200
pm.test("Status code is 200", function () {
    pm.response.to.have.status(200);
});

// Test 2: Check one of multiple valid status codes
pm.test("Status is 200 or 201", function () {
    pm.expect(pm.response.code).to.be.oneOf([200, 201]);
});

// Test 3: Response time check
pm.test("Response time < 500ms", function () {
    pm.expect(pm.response.responseTime).to.be.below(500);
});
JSON Response Assertions 
// Always parse JSON first
const response = pm.response.json();

// Check property value
pm.test("Status is OK", function () {
    pm.expect(response.status).to.eql("OK");
});

// Check nested property
pm.test("Has stock > 0", function () {
    pm.expect(response.currentStock).to.be.above(0);
});

// Check type
pm.test("Name is a string", function () {
    pm.expect(response.name).to.be.a("string");
});

// Bracket notation for special characters
response["current-stock"]  // when key has hyphens
Advanced Test Patterns 
// Check array + filter
const books = pm.response.json();
const available = books.filter(b => b.available);

pm.test("Has available books", () => {
    pm.expect(available.length).to.be.above(0);
});

// Partial object match
pm.test("Order contains required fields", () => {
    pm.expect(response).to.include({
        customerName: "John"
    });
});

// Check header exists & value
pm.test("Content-Type is JSON", () => {
    pm.expect(pm.response.headers.get("Content-Type"))
        .to.include("application/json");
});

// Debug: log to Postman Console
console.log("Response:", response);

Variable Scopes
Understanding scope hierarchy and when to use each type

Postman has four variable scopes, from broadest to narrowest. When you use pm.variables.get(), Postman looks from narrowest to broadest scope.

🌍 Global Variables
Available to ALL collections in a workspace. Good for quick prototyping or sharing data between collections.
pm.globals.set('key', value) / pm.globals.get('key')
🌐 Environment Variables
Tied to a selected environment (e.g., localhost, staging, production). Great for swapping base URLs and credentials.
pm.environment.set('key', value) / pm.environment.get('key')
📁 Collection Variables
Scoped to a single collection. Best default choice when you only have one environment. Stored in the collection file.
pm.collectionVariables.set('key', value) / pm.collectionVariables.get('key')
⚡ Local Variables
Only available in the current request. Automatically removed after execution. Used in collection runner.
pm.variables.set('key', value) / pm.variables.get('key')

Data Extraction & Chaining
Passing data between requests dynamically

Instead of hardcoding IDs and values, extract them from API responses and store in variables. This makes tests resilient to data changes.

Request Chaining Pattern 
// In "POST /orders" Tests tab — extract and save orderId
const response = pm.response.json();
const orderId = response.orderId;

pm.globals.set("orderId", orderId);
console.log("Saved orderId:", orderId);

// In "GET /orders/:orderId" — use the saved variable
// URL: {{baseUrl}}/orders/{{orderId}}

// Find first available non-fiction book
const books = pm.response.json();
const book = books.filter(
    b => b.available && b.type === "non-fiction"
)[0];

pm.globals.set("bookId", book.id);  // Use in next request

Random Data with Dynamic Variables

Use Postman's built-in dynamic variables to generate unique test data on each run:

{{$randomFullName}}
{{$randomEmail}}
{{$randomInt}}
{{$guid}}
{{$timestamp}}
{{$randomFirstName}}
{{$randomCompanyName}}
{{$randomUrl}}
Use in Scripts To use dynamic variables inside a pre-request or test script (not just the URL/body), use: pm.variables.replaceIn('{{$randomFullName}}')

Collection Runner
Running entire test suites automatically

The Collection Runner executes every request in a collection in order — all in one click. It shows pass/fail results for every test in every request.

▶️
Run All Requests
Execute the entire collection in sequence. Default order matches the collection layout.
💾
Save Responses
Check "Save responses" to inspect each response body after the run for debugging.
🔄
Iterations
Run the collection multiple times. Combine with data files for data-driven testing.

Controlling Execution Order

Request Flow Control 
// Jump to a specific request (by name)
pm.execution.setNextRequest("Get Order Details");

// Stop the collection run early
pm.execution.setNextRequest(null);

// Note: postman.setNextRequest() is DEPRECATED
// Use pm.execution.setNextRequest() instead

Newman CLI
Running Postman collections from the command line

Newman is a CLI tool that runs Postman collections outside the UI — perfect for CI/CD pipelines. It requires Node.js.

1
Install Newman
npm install -g newman — then verify with newman --version
2
Run a Collection
Export as JSON or use a public link. Then run: newman run collection.json
3
Generate HTML Reports
Install newman-reporter-htmlextra for rich, visual test reports.
Terminal — Newman Commands 
# Basic run
newman run collection.json

# With environment file
newman run collection.json -e environment.json

# With HTML report
newman run collection.json -r htmlextra \
  --reporter-htmlextra-export report.html

# Pass global variables
newman run collection.json \
  --global-var "baseUrl=https://api.example.com"

# Add delay between requests (ms)
newman run collection.json --delay 1000

# Fix weird unicode in CI
newman run collection.json --disable-unicode

Postman CLI GitHub Actions Forks & PRs
GitHub Actions Workflow (.github/workflows/api-tests.yml) 
name: API Tests
on: [push, pull_request]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      
      - name: Install Newman
        run: npm install -g newman
      
      - name: Run API Tests
        run: |
          newman run collection.json \
            --global-var "apiKey=${{ secrets.API_KEY }}" \
            -r cli,junit \
            --reporter-junit-export results.xml
      
      - name: Upload Results
        uses: actions/upload-artifact@v3
        with:
          name: test-results
          path: results.xml
Never hardcode secrets in CI files Always use GitHub repository secrets (Settings → Secrets) for API keys, tokens, and passwords. Reference them as ${{ secrets.YOUR_SECRET }}.

JSON Schema Validation
Validating response structure automatically

JSON Schema lets you define the shape of an expected response and automatically validate that every response matches it. This catches unexpected API changes early.

Schema Validation Test 
const schema = {
    "type": "object",
    "required": ["id", "bookId", "customerName"],
    "properties": {
        "id": { "type": "string" },
        "bookId": { "type": "integer" },
        "customerName": { "type": "string" },
        "createdBy": { "type": "string" },
        "quantity": { "type": "integer", "minimum": 1 },
        "timestamp": { "type": "string", "format": "date-time" }
    },
    // Block unexpected extra fields
    "additionalProperties": false
};

pm.test("Response matches schema", () => {
    pm.response.to.have.jsonSchema(schema);
});
Schema Type Reference Valid types: string, number, integer, boolean, null, object, array

Mock Servers
Testing without a real API backend

Mock servers let you create simulated API endpoints with predefined responses. Use them to test schema validation, error handling, or when the real API isn't ready yet.

🧪
Safe Schema Testing
Test schema changes without affecting the real API or creating real data.
🚧
Early Development
Start writing and testing against a mock before the real backend is built.
🔁
Controlled Responses
Return exactly the response you want to test edge cases: missing fields, wrong types, errors.
